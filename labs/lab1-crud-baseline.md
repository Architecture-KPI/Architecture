# Лабораторна робота №1: CRUD — «зроби як вмієш»

## Мета

Отримати працюючий baseline. Ви пишете так, як звикли, без нав'язаних патернів. Цей код стане відправною точкою для рефакторингу в наступних лабораторних.

## Домен

Будь-який на ваш вибір (будь-яка мова, будь-який фреймворк).

## Завдання

### 1. Описати юзкейси системи

Перед написанням коду необхідно формалізувати, що саме система має робити. Скласти перелік юзкейсів (use cases) для вашого домену.

Для кожного юзкейсу вказати:
- **Назва** (наприклад, «Створення замовлення»)
- **Актор** — хто ініціює (клієнт, адміністратор)
- **Передумови** — що має бути виконано до початку
- **Основний сценарій** — кроки успішного виконання
- **Альтернативні сценарії / помилки** — що відбувається при невалідних даних, конфліктах тощо

Юзкейси розмістити в `docs/use-cases.md` у репозиторії.

### 2. Реалізувати проект для обраного домену

- REST API з коректними HTTP-методами та статус-кодами
- Обробка помилок: коректні HTTP-статуси (400 для невалідних даних, 404 для неіснуючих ресурсів, 409 для конфліктів)
- Для створення та оновлення сутностей **обов'язкові інваріанти** — правила, які завжди повинні виконуватись. Приклади:
  - email повинен бути валідним
  - часовий слот не може бути в минулому
  - часовий діапазон має бути коректним (початок < кінець)
  - інші правила, специфічні для вашого домену

### 3. Додати автентифікацію

- Реалізувати реєстрацію та вхід користувача
- Механізм на ваш вибір: cookies або JWT
- Захистити ендпоінти — неавторизовані запити повертають `401`

### 4. Написати тести (піраміда тестування)

- **Unit-тести**: логіка валідації, перевірка бізнес-правил
- **Integration-тести**: тести ендпоінтів (HTTP-запит → відповідь), перевірка що API працює коректно з базою даних

## Критерії оцінювання

| Критерій | На що звернути увагу |
|----------|---------------------|
| Функціональність | Усі операції працюють, бізнес-правила (інваріанти) виконуються |
| Автентифікація | Є реєстрація, вхід, захист ендпоінтів; неавторизовані запити повертають 401 |
| Юзкейси | Покривають основні сценарії, описані актори, передумови, альтернативні шляхи |
| Тести | Є unit і integration тести, покривають основні сценарії та edge cases |
| CI | Pipeline налаштований, тести проходять автоматично |
| README | Проєкт можна запустити за інструкцією |

## Питання для самоаналізу

Після виконання роботи дайте відповідь на ці питання (письмово, 1–2 абзаци на кожне):

### Архітектура

1. **Де живе бізнес-логіка?** Подивіться на ваш код — валідація, перевірка інваріантів, прийняття рішень відбувається в контролерах, сервісах чи окремих класах?
2. **Як пов'язані моделі БД і бізнес-логіка?** Чи використовуєте ви ORM-моделі безпосередньо для бізнес-правил, чи є окремі об'єкти?
3. **Наскільки легко було б замінити базу даних?** Уявіть, що вам потрібно перейти з PostgreSQL на MongoDB — скільки файлів доведеться змінити?
4. **Наскільки складно було тестувати?** Чи залежать ваші unit-тести від бази даних або фреймворку? Чи можна запустити тести бізнес-логіки без підняття інфраструктури?

### Масштабування

5. **Наскільки легко було б масштабувати ваш сервіс?** Уявіть, що навантаження зросло в 100 разів — які місця зламаються першими? Чи є у вашому коді стан (in-memory кеші, глобальні змінні), який заважає запустити кілька інстансів? Які архітектурні рішення ви б змінили?
