# Лабораторна робота 3: CQS (Command-Query Separation)

## Мета

Розділити операції запису (команди) та читання (запити) на рівні Application Layer. Студенти бачать, що читання і запис мають різні вимоги до коду, тестування та оптимізації.

## Завдання

### 1. Виділити команди (Commands) та їхні обробники (Command Handlers)

Команда — об'єкт-намір, що описує дію, яка змінює стан системи. Кожна команда обробляється одним `CommandHandler`.

Студент сам визначає набір команд, виходячи з юзкейсів із лабораторної 1. Кожна операція запису (створення, оновлення, видалення, зміна статусу) — це окрема команда.

Вимоги до команд:
- Команда — простий об'єкт з даними (DTO), **без логіки**
- Handler проходить через домен, валідує інваріанти
- Команда **не повертає дані** (допустимий виняток — ID створеної сутності)

### 2. Виділити запити (Queries) та їхні обробники (Query Handlers)

Запит — об'єкт, що описує, які дані потрібно отримати. Не змінює стан системи.

Студент сам визначає набір запитів, виходячи з юзкейсів із лабораторної 1. Кожна операція читання (отримання списку, пошук, фільтрація, отримання деталей) — це окремий запит.

Вимоги до запитів:
- Запити можуть читати **напряму з БД**, минаючи доменний шар
- Запити повертають **Read Models** (DTO), оптимізовані під потреби клієнта — не доменні моделі
- Запити **не мутують стан**


### 3. Рефакторинг Presentation Layer

Контролери стають «тонкими» — їхня єдина задача: маппінг HTTP-запиту в Command або Query та передача відповідному Handler.

До (лаба 2):
```
Controller → UseCase / ApplicationService → Domain → Repository
```

Після (лаба 3):
```
Controller → DTO  → CommandHandler → Domain → Repository
Controller → DTO   → QueryHandler
```

Вимоги:
- Контролери **не містять логіки** — лише маппінг HTTP → Command / Query

### 7. Тестування

Команди та запити тестуються **по-різному**:

Тести команд:
- поведінкові ( тести бізнес логіки )

**Тести запитів** 
- інтеграційно ендпоінт

Студент має **проаналізувати**: що стало простіше тестувати порівняно з лабою 2, що складніше?

### 8. Написати аналіз

Порівняльний аналіз (1-2 сторінки), що відповідає на питання:

- Що змінилося в структурі проєкту порівняно з лабораторною 2?
- Переваги CQS: чим допомогло розділення?
- Недоліки CQS: що ускладнилося?
- Чим підхід з Command / Query Handler відрізняється від Service, який робить все сам?
- Як CQS впливає на розширюваність? (додати нову команду vs змінити існуючий сервіс)

Розмістити в `docs/analysis/lab3.md`.

## Критерії оцінювання

| Критерій | На що звернути увагу                                                         |
|----------|------------------------------------------------------------------------------|
| Команди | Не повертають дані (або лише ID), проходять через домен                      |
| Запити | Не мутують стан, повертають DTO                                              |
| Handlers | Маленькі, з однією відповідальністю, делегують домену                        |
| Контролери | Містять лише маппінг HTTP → Command / Query                                  |
| Тестування | Різний підхід до тестування команд і запитів                                 |
| Аналіз | Конкретне порівняння з лабою 2, плюси та мінуси CQS                          |
