# Лабораторна робота 4: Синхронна та асинхронна комунікація між компонентами

## Мета

Навчитися організовувати взаємодію між компонентами системи. Студенти реалізують як синхронну, так і асинхронну комунікацію, порівнюють підходи та обирають оптимальний для кожного сценарію.

## Теоретичні матеріали

Перед виконанням лабораторної опрацюйте:

- [Патерни синхронної та асинхронної комунікації](../docs/communication-patterns.md)
- [Доменні події та Eventual Consistency](../docs/domain-events.md) (розділи про доменні події та Event Bus)

## Контекст

До цього моменту вся логіка системи виконується «в лінію»: контролер → handler → домен → репозиторій. Але в реальних системах є побічні операції, які мають відбуватися *у відповідь* на бізнес-дію — нотифікації, аудит, аналітика, інтеграція із зовнішніми системами тощо.

У цій лабораторній студенти виділяють **допоміжний компонент** (який саме — визначає студент, виходячи зі свого домену) і реалізують два способи комунікації з ним: синхронний та асинхронний.

## Завдання

### 1. Визначити побічні операції та виділити компонент

Проаналізувати юзкейси системи та визначити:

- Які побічні операції мають відбуватися у відповідь на основні бізнес-дії? (нотифікація, аудит, аналітика, синхронізація з іншою системою тощо)
- Який допоміжний компонент відповідає за ці операції?

Вимоги до компонента:

- Логічно окремий модуль з **власною відповідальністю** (не частина основного Handler)
- Має чітко визначений **контракт (інтерфейс)**: які повідомлення приймає та що з ними робить
- Знаходиться в окремій папці / пакеті в структурі проєкту

### 2. Реалізувати синхронну комунікацію

Перший варіант — Command Handler після основної операції **напряму викликає** допоміжний компонент:

- Виклик відбувається в тому ж потоці виконання
- Якщо допоміжний компонент падає — потрібно вирішити: основна операція відкочується чи помилка ігнорується?
- Допоміжний компонент впроваджується через інтерфейс (DIP зберігається)

Зафіксувати спостереження:

- Час відповіді API (основна операція + побічна)
- Поведінка системи при збої допоміжного компонента
- Чи знає основний Handler про конкретну реалізацію допоміжного компонента?

### 3. Реалізувати асинхронну комунікацію

Другий варіант — Command Handler після основної операції публікує **доменну подію**, допоміжний компонент **підписується** на неї:

- Реалізувати механізм доставки подій: **in-process Event Bus** або зовнішній брокер (RabbitMQ, Redis тощо) — на вибір студента
- Command Handler публікує подію після успішного завершення основної операції
- Допоміжний компонент підписується на потрібні події та обробляє їх **незалежно**
- Основна операція завершується одразу, не чекаючи на обробку побічних ефектів

Вимоги до доменних подій:

- Іменуються в **минулому часі** (щось вже відбулося)
- **Незмінні** (immutable) — подія описує факт
- Містять **достатньо контексту**, щоб підписник міг обробити подію без додаткових запитів

### 6. Написати аналіз

Порівняльний аналіз (1-2 сторінки), що відповідає на питання:

- Що змінилося порівняно з лабораторною 3?
- Які побічні операції ви виділили і чому?
- Порівняння синхронного та асинхронного підходів:
  - Час відповіді API
  - Поведінка при збоях
  - Зв'язаність між компонентами
  - Складність реалізації та тестування
- Який підхід ви б обрали для production-системи і чому?

Розмістити в `docs/analysis/lab4.md`.

## Що здати


| Артефакт               | Опис                                             |
| ---------------------- | ------------------------------------------------ |
| Допоміжний компонент   | Окремий модуль з власним контрактом              |
| Синхронна комунікація  | Прямий виклик через інтерфейс, обробка помилок   |
| Асинхронна комунікація | Доменні події + Event Bus / брокер, підписники   |
| Аналіз                 | `docs/analysis/lab4.md` — порівняння підходів    |


## Критерії оцінювання


| Критерій            | На що звернути увагу                                                      |
|---------------------| ------------------------------------------------------------------------- |
| Розділення          | Допоміжний компонент — окремий модуль, не вбудований у основний Handler   |
| Контракт            | Компонент має чітко визначений інтерфейс                                  |
| Синхронний варіант  | Працює, обробка помилок продумана                                         |
| Асинхронний варіант | Основна операція не залежить від підписників, події доставляються         |
| Події               | Іменовані в минулому часі, незмінні, містять достатньо контексту          |
| Тестування          | Обидва варіанти покриті, тести перевіряють різну поведінку при збоях      |
| Аналіз              | Конкретне порівняння з реальними спостереженнями (час, збої, зв'язаність) |
