## Завдання

### 1. Виділити доменні моделі

Створити доменні моделі, які **не залежать від бази даних**:

- Модель БД (Entity / ORM-модель) ≠ доменна модель — це різні класи з різними відповідальностями
- Обрати підхід: **Anemic** або **Rich Domain Model** (рішення зафіксувати в ADR з обґрунтуванням)
- Реалізувати **маппінг** між доменними моделями та моделями БД (маппер живе в Infrastructure Layer)

Студент сам визначає набір доменних моделей та їхні інваріанти, виходячи з юзкейсів, описаних у лабораторній 1. Зверніть увагу на:
- Які сутності є в системі та яка їхня ключова поведінка?
- Які інваріанти мають бути захищені? (валідність полів, бізнес-правила, обмеження)
- Чи є Value Objects — об'єкти без ідентичності, що описують концепцію (наприклад, часовий діапазон, email, грошова сума)?

### 2. Реалізувати Domain Factory

Фабрика відповідає за створення доменних об'єктів із перевіркою інваріантів:

- **Прості інваріанти** (не потребують БД): слот не в минулому, email валідний, часовий діапазон коректний
- **Складні інваріанти** (потребують БД): унікальність email, перетин слотів — фабрика приймає інтерфейс репозиторію через DIP
- Якщо інваріант порушено — фабрика кидає доменну помилку (не HTTP-помилку, не виняток фреймворку)

### 3. Виділити 4 шари

Рефакторити проєкт із лаби 1 у чітку шарову архітектуру:

| Шар | Відповідальність | Залежить від |
|-----|-----------------|--------------|
| **Presentation** | Контролери, DTO, маппінг HTTP ↔ Application | Application |
| **Application** | Use Cases, оркестрація (домен + інфраструктура через інтерфейси) | Domain |
| **Domain** | Бізнес-логіка, доменні моделі, інтерфейси репозиторіїв | Нічого |
| **Infrastructure** | Реалізація репозиторіїв, ORM, зовнішні сервіси, маппери | Domain |

Вимоги:
- Кожен шар — **окремий пакет / модуль** у структурі проєкту
- Дотримуватись **DIP**: домен визначає інтерфейси (порти), інфраструктура їх реалізує (адаптери)
- **Правило залежностей**: `Presentation → Application → Domain ← Infrastructure`
- Домен **не імпортує** нічого з інших шарів (жодних ORM, HTTP, фреймворків)

Очікувана структура проєкту (приклад):
```
src/
├── presentation/
│   ├── controllers/
│   └── dto/
├── application/
│   └── use_cases/
├── domain/
│   ├── entities/
│   ├── aggregates/
│   ├── value-objects/
│   ├── factories/
│   └── repositories/    # інтерфейси репозиторіїв
└── infrastructure/
    ├── repositories/    # реалізації репозиторіїв
    ├── orm_models/      # ORM-сутності
    └── mappers/         # доменна модель ↔ ORM-модель
```

### 4. Створити діаграму архітектури

Намалювати діаграму, що відображає **шари системи та залежності між ними**:

- 4 шари з ключовими компонентами кожного
- Стрілки залежностей (хто від кого залежить)
- Позначити, де живуть інтерфейси і де — їх реалізації (DIP)

Формат: Mermaid у `.md` файлі або зображення. Розмістити в `docs/architecture.md`.

### 5. Рефакторинг тестів

Тести мають відображати нову архітектуру:

**Unit-тести домену:**
- Тестують бізнес-логіку **без БД, без фреймворку** — чисті доменні моделі та фабрики
- Мокаємо репозиторії через інтерфейси (фейкові in-memory реалізації або моки)
- Студент повинен **продемонструвати**, що доменні тести запускаються без підняття БД і без HTTP-сервера

**Unit-тести Application Layer:**
- Тестують use cases
- Репозиторії мокаються через інтерфейси

**Integration-тести:**
- Тести ендпоінтів (HTTP-запит → відповідь), повний цикл через усі шари
- Працюють з реальною БД (або тестовою)

### 6. Написати аналіз

Порівняльний аналіз (1-2 сторінки), що відповідає на питання:
- Що змінилося в структурі проєкту порівняно з лабораторною 1?
- Які переваги дало розділення на шари?
- Які недоліки / ускладнення з'явилися? (більше коду, маппінг, складніша навігація)
- Наскільки простіше тепер змінити БД або фреймворк?
- Чому було обрано Anemic / Rich Domain Model? (посилання на ADR)

Розмістити в `docs/analysis/lab2.md` у репозиторії.

## Критерії оцінювання

| Критерій | На що звернути увагу |
|----------|---------------------|
| Залежності | Немає зворотних залежностей (домен не імпортує ORM, HTTP, фреймворк) |
| Доменні моделі | Дійсно відрізняються від моделей БД, містять бізнес-логіку або інваріанти |
| Domain Factory | Перевіряє інваріанти, приймає залежності через інтерфейси |
| Маппінг | Є маппери між шарами, живуть в Infrastructure |
| Структура проєкту | 4 окремі пакети/модулі, відповідають правилу залежностей |
| Діаграма | Коректно відображає шари, залежності та DIP |
| Тести домену | Працюють без БД та фреймворку |
| ADR | Рішення Anemic vs Rich обґрунтоване, альтернативи розглянуті |
| Аналіз | Конкретне порівняння з лабою 1, не загальні фрази |
