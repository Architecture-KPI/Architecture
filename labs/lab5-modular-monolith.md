# Лабораторна робота 5: Модульний моноліт

## Мета

Розділити систему на ізольовані модулі (bounded contexts) в рамках одного розгортуваного артефакту. Студенти виділяють контексти, запроваджують ACL між ними та організовують міжмодульну комунікацію через доменні події з eventual consistency.

## Контекст

На попередніх лабораторних студенти побудували систему з чітким розділенням на шари, CQS та асинхронною комунікацією. Але вся система — це один великий контекст: усі моделі, репозиторії та handlers живуть разом і мають доступ одне до одного.

У цій лабораторній студенти виділяють **обмежені контексти** (bounded contexts) і перетворюють систему на модульний моноліт — кожен модуль має чіткі межі, власну модель та публічний контракт.

Студенти працюють у рамках **існуючих моделей** своєї системи — конкретна доменна область не нав'язується.

## Завдання

### 1. Визначити обмежені контексти (Bounded Contexts)

Виділити **два обов'язкові контексти**: основний модуль системи та аналітичний модуль.

**Основний контекст (Core):**

Існуюча бізнес-логіка системи — все, що було реалізовано в попередніх лабораторних. Оформлюється як окремий модуль з публічним контрактом.

**Аналітичний контекст (Analytics):**

Окремий модуль, який підходить для будь-якої предметної області та природно демонструє ключові концепції:

- **Власна модель даних** — агрегати, метрики, проєкції, а не ті самі сутності, що в основному домені
- **Read-only consumer** — споживає дані з основного контексту, але не змінює їх
- **Зв'язок через події** — основний контекст публікує events, аналітика підписується та будує свої проєкції
- **ACL** — аналітика маппить вхідні події у свою внутрішню модель, не залежачи від структури основного домену
- **Eventual consistency** — дані в аналітиці відстають від основного контексту, і це нормально та by design

**Додатковий контекст (опціонально, на вибір студента):**

За бажанням студент може виділити ще один контекст зі свого домену. Це не обов'язково, але буде враховано при оцінюванні. Запитання для аналізу:
- Які групи сутностей та операцій логічно належать разом?
- Де одна й та ж сутність має різний зміст у різних частинах системи?

Вимоги:
- Кожен контекст має мати **власну відповідальність**, що не перетинається з іншими
- Обґрунтувати вибір меж: чому основний + аналітика, що дає таке розділення
- Зафіксувати рішення в ADR

### 2. Створити модульну структуру

Перетворити проєкт на набір ізольованих модулів:

- Кожен модуль — **окрема папка** з повною внутрішньою архітектурою (domain, application, infrastructure)
- Кожен модуль має **публічний контракт** — єдина точка входу для інших модулів (набір DTO та методів, доступних ззовні)
- **Прямий доступ** до внутрішніх класів іншого модуля **заборонений**

Очікувана структура (приклад):
```
src/
├── modules/
│   ├── module_a/
│   │   ├── api.py              # публічний контракт модуля
│   │   ├── domain/
│   │   ├── application/
│   │   └── infrastructure/
│   ├── module_b/
│   │   ├── api.py
│   │   ├── domain/
│   │   ├── application/
│   │   └── infrastructure/
│   └── module_c/
│       ├── api.py
│       ├── domain/
│       ├── application/
│       └── infrastructure/
├── shared/
│   └── events/                 # спільні контракти подій
└── presentation/
    └── controllers/
```

### 3. Реалізувати ACL (Anti-Corruption Layer)

Між модулями, що спілкуються, створити антикорупційний шар:

- ACL **транслює** зовнішню модель (з публічного контракту іншого модуля) у внутрішню модель споживача
- Кожен модуль працює **лише зі своїми** внутрішніми моделями
- ACL належить **споживачу** — модулю, який отримує зовнішні дані

Що має бути:
- Клас-транслятор для кожного зв'язку між модулями
- Внутрішні моделі, в які транслюються зовнішні дані
- Зовнішня модель **не проникає** далі ACL

### 4. Організувати міжмодульну комунікацію через доменні події

Розширити механізм подій з лабораторної 4 для комунікації між модулями:

- Визначити ключові доменні події, які модулі публікують після бізнес-операцій
- Інші модулі підписуються на потрібні події та реагують через свої handlers
- Реалізувати **eventual consistency**: основна операція завершується одразу в межах свого модуля, побічні ефекти в інших модулях обробляються асинхронно

Студент має визначити:
- Де у системі **strong consistency** (в межах одного модуля)?
- Де **eventual consistency** (між модулями)?
- Чому такий розподіл прийнятний для бізнесу?

### 5. Тестування

**Unit-тести модулів:**
- Кожен модуль тестується **ізольовано** — без залежності від інших модулів
- Зовнішні залежності (інші модулі) мокаються через їхній публічний контракт

**Integration-тести міжмодульної комунікації:**
- Перевірка, що подія опублікована одним модулем коректно обробляється іншим
- Перевірка, що ACL правильно транслює моделі
- Перевірка eventual consistency: побічний ефект відбувається після основної операції

### 6. Оновити діаграму архітектури

Оновити `docs/architecture.md` — фінальна діаграма, що відображає:
- Модулі (bounded contexts) як окремі блоки
- Публічні контракти модулів
- ACL між модулями
- Потоки подій між модулями

### 7. Написати фінальний аналіз

Це остання лабораторна — аналіз має бути **ретроспективним** (2-3 сторінки):

**Аналіз лабораторної 5:**
- Як ви визначили межі контекстів? Які альтернативи розглядали?
- Як ACL захищає модулі від змін в інших модулях?
- Де у системі strong consistency, а де eventual consistency? Чому це прийнятно?
- Що зміниться, якщо потрібно буде виділити один з модулів у окремий сервіс?

**Ретроспектива курсу (від лаби 1 до лаби 5):**
- Як змінювалася архітектура проєкту на кожному етапі?
- Які архітектурні рішення виявилися найціннішими?
- Що було б зроблено інакше, знаючи кінцевий результат?
- Які trade-offs ви побачили між простотою та гнучкістю?

Розмістити в `docs/analysis/lab5.md`.

## Критерії оцінювання

| Критерій | На що звернути увагу |
|----------|---------------------|
| Межі контекстів | Обґрунтовані, не штучний поділ заради поділу |
| Ізоляція модулів | Модулі не імпортують внутрішні класи один одного |
| Публічний контракт | Визначений явно, достатній для потреб інших модулів |
| ACL | Є трансляція моделей, зовнішня модель не проникає далі ACL |
| Доменні події | Іменовані в минулому часі, містять контекст, обробляються асинхронно |
| Eventual consistency | Студент розуміє, де strong, а де eventual, і чому |
| Тестування | Модулі тестуються ізольовано, міжмодульна комунікація покрита |
| ADR | Вибір bounded contexts обґрунтований, альтернативи розглянуті |
| Ретроспектива | Чесний аналіз еволюції від лаби 1 до лаби 5, конкретні спостереження |
