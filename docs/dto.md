## Вступ

Запитайте п'ятьох розробників, що таке DTO — і отримаєте сім різних відповідей. Хтось скаже, що це «модель для API». Інший — що «клас без логіки». Третій назве DTO все підряд: ORM-сутність, request body, response, конфіг. В результаті DTO перетворюється на розмите поняття, яке нічого не означає.

А між тим DTO — це чітко визначений патерн з конкретною метою. Він з'явився не тому, що комусь було нудно створювати зайві класи. Він вирішує реальну проблему, і якщо розуміти яку — стає очевидно, де він потрібен, а де ні.

## Що таке DTO і яку проблему він вирішує

DTO (Data Transfer Object) — це об'єкт, єдина мета якого — **перенести дані через межу** між двома шарами або компонентами системи.

### Проблема

Уявіть: контролер отримує HTTP-запит на створення бронювання. Що він робить далі? Передає дані в Application Layer. Але в якому вигляді?

Варіант 1 — передати сирий `dict` / `JSON`:

```python
class BookingController:
    def create(self, request):
        data = request.json  # dict з невідомою структурою
        self.handler.handle(data)  # що всередині? які поля? які типи?
```

Проблеми: немає типізації, немає контракту, будь-яка друкарська помилка в ключі — і баг у рантаймі.

Варіант 2 — передати доменну модель:

```python
class BookingController:
    def create(self, request):
        booking = Booking(...)  # створюємо доменний об'єкт прямо в контролері
        self.handler.handle(booking)
```

Проблеми: контролер знає про домен, форма HTTP-запиту прив'язана до доменної моделі. Додали поле в домен — зламався API. Перейменували поле в API — зламався домен.

Варіант 3 — DTO:

```python
@dataclass(frozen=True)
class CreateBookingRequest:
    user_id: str
    resource_id: str
    start_time: str  # ISO 8601 рядок з HTTP
    end_time: str

class BookingController:
    def create(self, request):
        dto = CreateBookingRequest(**request.json)
        self.handler.handle(dto)
```

DTO — це **контракт на кордоні**. Він каже: «ось які дані я очікую, ось їхні типи, нічого зайвого». Він не знає про домен. Домен не знає про нього. Кожен бік кордону може змінюватися незалежно.

### Визначення

DTO — це:
- Об'єкт **без поведінки** (тільки дані)
- Існує **на межі** між шарами або компонентами
- Визначає **контракт** передачі даних
- **Не є** доменною моделлю, ORM-сутністю або конфігом

## Типи DTO та де вони живуть

DTO з'являється всюди, де є кордон. У шаровій архітектурі є кілька таких кордонів:

### Request DTO (вхідний)

Описує дані, які приходять ззовні в систему (HTTP-запит, повідомлення з черги, CLI-аргументи).

```python
@dataclass(frozen=True)
class CreateBookingRequest:
    user_id: str
    resource_id: str
    start_time: str
    end_time: str
```

**Де живе**: Presentation Layer (або на вході Application Layer — залежить від того, хто визначає контракт).

### Response DTO (вихідний)

Описує дані, які система повертає назовні.

```python
@dataclass(frozen=True)
class BookingResponse:
    id: str
    user_name: str       # денормалізовано — клієнту зручніше
    resource_name: str   # ніж отримувати окремо user і resource
    start_time: str
    end_time: str
    status: str
```

**Де живе**: Presentation Layer (або Application Layer, якщо це Read Model для CQS).


## DTO та валідація

Валідація — ще одне джерело плутанини. «Де валідувати — в DTO, в контролері, в домені?» Відповідь: **на кожному рівні своя валідація**.

### Рівні валідації

| Рівень | Що перевіряє | Де відбувається | Приклади |
|--------|-------------|----------------|---------|
| **Формат** | Чи дані взагалі коректного типу/формату | Presentation Layer (Request DTO) | Чи `start_time` — валідний ISO 8601? Чи `email` — рядок з `@`? Чи обов'язкові поля присутні? |
| **Бізнес-правила** | Чи дані мають сенс для бізнесу | Domain Layer (моделі, фабрики) | Чи `start_time < end_time`? Чи слот не в минулому? Чи email унікальний? |

### Валідація формату (в DTO / Presentation)

Це перевірка: «чи дані фізично валідні?» — до того, як вони потраплять глибше в систему.


### Бізнес-валідація (в домені)

Це перевірка: «чи операція має сенс?» — з точки зору бізнес-правил.

```python
class BookingFactory:
    def create(self, user_id, resource_id, start, end):
        time_slot = TimeSlot(start, end)
        if time_slot.is_in_past():
            raise DomainError("Не можна бронювати час у минулому")
        if time_slot.duration_minutes() < 30:
            raise DomainError("Мінімальна тривалість — 30 хвилин")
        # ...
```

### Чому не валідувати все в одному місці?

Якщо валідацію формату перенести в домен — домен почне знати про формати HTTP-запитів (ISO 8601 рядки, JSON-структури). Якщо бізнес-правила перенести в DTO — DTO стане «товстим» і залежатиме від репозиторіїв.

Кожен рівень валідує **те, за що відповідає**, і не більше.

## Як DTO перетинає кордони

Ключове правило: **DTO не проходить наскрізь через усі шари**. На кожному кордоні може бути свій DTO.

### Чому не один DTO на все?

Спокуса: створити один `BookingDTO` і використовувати скрізь — у контролері, в handler'і, у відповіді. Це швидко, менше коду.

Проблеми:
- Зміна API-формату ламає handler
- Handler починає залежати від структури HTTP-запиту
- Response містить поля, які не потрібні клієнту (або навпаки — не вистачає полів)
- Неможливо мати різні представлення для різних клієнтів (мобільний додаток vs веб)


## Поширені міфи

### «DTO — це будь-який клас без логіки»

Ні. Клас без логіки може бути конфігом, value object, entity без поведінки. DTO — це конкретно об'єкт **для передачі даних через кордон**. Якщо об'єкт не перетинає кордон — це не DTO, навіть якщо він «просто дані».

### «Доменна модель може бути DTO»

Ні. Якщо доменна модель повертається як HTTP-відповідь — зовнішній контракт (API) зв'язується з внутрішньою структурою (домен). Перейменували поле в домені — зламався API у клієнтів. Це саме те, від чого DTO захищає.

### «DTO — це зайвий бойлерплейт»

Маппінг — це ціна за ізоляцію. Без нього шари зливаються: зміна в одному місці каскадно ламає інші. На маленькому проєкті це не відчутно. На проєкті, що росте — це рятує.

### «Один DTO на сутність — достатньо»

Різні кордони мають різні потреби. Request DTO містить те, що надсилає клієнт. Response DTO — те, що клієнт хоче бачити (можливо, з денормалізованими даними). Command DTO — те, що потрібно handler'у (можливо, вже з розпарсеними типами). Це різні контракти.

### «Валідація — в DTO»

Тільки валідація **формату**. Бізнес-правила — в домені. DTO не знає про бізнес, домен не знає про формати HTTP.
