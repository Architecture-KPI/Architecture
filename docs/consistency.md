## Вступ

Два користувачі одночасно бронюють один і той самий слот. Обидва бачать його як вільний. Обидва натискають «підтвердити». Один з них отримає помилку — або, якщо система погано спроєктована, обидва отримають підтвердження, а потім хтось побачить подвійне бронювання.

Це питання **консистентності** — наскільки система гарантує, що дані, які бачать різні учасники, є актуальними, узгодженими і не суперечать одне одному.

---

## Що таке консистентність

Консистентність — це гарантія того, що **дані в системі відповідають визначеним правилам (інваріантам)** у будь-який момент часу, або принаймні після завершення операції.

В контексті баз даних це частина ACID:

| Властивість | Значення |
|-------------|----------|
| **A**tomicity | Транзакція або виконується повністю, або не виконується зовсім |
| **C**onsistency | Після транзакції дані залишаються у валідному стані |
| **I**solation | Паралельні транзакції не впливають одна на одну |
| **D**urability | Закомічені дані не втрачаються |

Консистентність у ACID означає: якщо до транзакції інваріанти виконувались, то після транзакції вони теж виконуються. Наприклад: «два бронювання не можуть займати один слот» — якщо це було правдою до INSERT, це має бути правдою і після.

## Рівні консистентності

### Strong Consistency

Після запису **всі** читачі одразу бачать нові дані. Кожне читання повертає результат останнього запису.

```
Client A: write(x = 5)    ────────> OK
Client B:                  read(x)  ────────> 5 ✓
```

Гарантується однією БД з транзакціями. Проста модель — але обмежує масштабування, бо вимагає координації між усіма вузлами.

**Коли потрібна:**
- Перевірка доступності слоту перед бронюванням — інакше подвійне бронювання
- Списання коштів — інакше подвійне списання
- Зміна паролю — інакше старий пароль ще працює

### Eventual Consistency

Після запису дані **стануть** консистентними через деякий час. Між записом і моментом, коли всі читачі побачать оновлення, є вікно неконсистентності.

```
Client A: write(x = 5)    ────────> OK
Client B:                  read(x)  ────────> 3 (стара версія)
           ...50ms пізніше...
Client B:                  read(x)  ────────> 5 ✓
```

Система **тимчасово** показує неактуальні дані, але з часом приходить до узгодженого стану.

**Коли прийнятна:**
- Надсилання email після бронювання — затримка в секунди некритична
- Оновлення аналітики — звіти можуть відставати
- Оновлення кешу — допустимо показати stale дані на кілька секунд
- Синхронізація між модулями через [події](events.md)

### Causal Consistency

Проміжний варіант: якщо операція B **залежить** від операції A, то всі побачать їх у правильному порядку. Незалежні операції можуть бути видні в довільному порядку.

```
Client A: write(x = 5)
Client A: write(y = x + 1)   — y залежить від x
Client B: read(y) = 6        — бачить y
Client B: read(x) = 5 ✓      — гарантовано бачить x теж
```

## CAP Theorem

### Формулювання

У розподіленій системі неможливо одночасно гарантувати всі три властивості:

| Властивість | Значення |
|-------------|----------|
| **C**onsistency | Кожне читання повертає останній запис (strong consistency) |
| **A**vailability | Кожен запит отримує відповідь (навіть якщо дані неактуальні) |
| **P**artition tolerance | Система працює, навіть якщо мережа між вузлами розірвана |

**Partition tolerance обов'язкова** в будь-якій розподіленій системі — мережа ламається завжди. Тому реальний вибір: **CP або AP**.

### CP: Consistency + Partition tolerance

Система відмовляє у відповіді, якщо не може гарантувати актуальність даних. Краще не відповісти, ніж відповісти неправильно.

**Приклад:** банківська система. Якщо вузол не може зв'язатися з іншими — краще відхилити транзакцію, ніж дозволити подвійне списання.

### AP: Availability + Partition tolerance

Система завжди відповідає, навіть якщо дані можуть бути неактуальними. Конфлікти вирішуються пізніше.

**Приклад:** DNS, кеші, соціальні мережі. Якщо лайк не одразу видно всім — це прийнятно. Головне — система відповідає.

### На практиці

CAP — це не бінарний вибір для всієї системи. **Різні частини** системи можуть мати різні гарантії:

| Операція | Вибір | Чому |
|----------|-------|------|
| Створення бронювання | CP | Подвійне бронювання неприпустиме |
| Стрічка активності | AP | Затримка в кілька секунд прийнятна |
| Баланс рахунку | CP | Подвійне списання неприпустиме |
| Кеш профілю | AP | Stale дані некритичні |

## Консистентність у контексті архітектури

### В межах одного агрегату — strong

Один [агрегат](entities-and-aggregates.md) змінюється в одній транзакції. Інваріанти перевіряються до коміту. Це strong consistency за замовчуванням:

```python
class Booking:
    def confirm(self) -> None:
        if self._status != "created":
            raise DomainError("Can only confirm created bookings")
        self._status = "confirmed"

# одна транзакція — save + commit
repo.save(booking)
uow.commit()
```

### Між модулями — eventual

Коли один модуль публікує [подію](events.md), інші обробляють її асинхронно. Між комітом основної операції і обробкою підписниками є затримка:

```
Booking Module: save() + commit()  ──> BookingCreated event
                                              │
Notification Module: ─────── 50ms ───── handle(event) ──> email sent
Analytics Module:    ────── 100ms ───── handle(event) ──> stats updated
```

Між комітом і обробкою система **тимчасово неконсистентна**. Це нормально і by design.

### Правило

**В межах одного агрегату** — strong consistency (одна транзакція). **Між модулями / агрегатами** — eventual consistency (через події). Це дозволяє модулям бути незалежними, при цьому зберігаючи інваріанти там, де це критично.

## Поширені міфи

### «Eventual consistency — це баг, який потрібно виправити»

Це не баг, а **архітектурне рішення**. Ви свідомо обмінюєте негайну консистентність на слабку зв'язаність і відмовостійкість. Для більшості побічних ефектів (email, аналітика, аудит) це виграшна угода.

### «Strong consistency потрібна завжди»

Ні. Strong consistency вимагає координації, що обмежує масштабування і збільшує латентність. Для аналітики, кешів, нотифікацій — eventual достатньо і дешевше.

### «CAP каже: обери два з трьох»

CAP формулювання спрощене. На практиці: partition tolerance обов'язкова, вибір між C і A робиться **per operation**, а не для всієї системи. Одна й та сама система може бути CP для платежів і AP для стрічки активності.

### «Якщо у мене одна БД — CAP не стосується»

CAP стосується розподілених систем. Якщо у вас одна PostgreSQL без реплік — ви маєте strong consistency і не потребуєте CAP-компромісів. CAP з'являється, коли додаєте репліки, кеші, інші сервіси або модулі з окремими сховищами.
