# Лабораторні роботи: Еволюційна архітектура ПЗ

## Про курс

Курс побудований за принципом **еволюційної архітектури**: кожна лабораторна робота продовжує попередню. Студенти працюють з одним проєктом протягом усього курсу і на практиці бачать, як архітектура змінюється разом із вимогами — від простого CRUD до модульного моноліту з подієво-орієнтованою комунікацією.

## Домен

**Система бронювання** (клініка / готель / коворкінг — на вибір студента).

Домен складніший за типовий Shop: є часові слоти, конфлікти розкладу, статуси бронювання (`створено → підтверджено → скасовано`), різні ролі (клієнт, адміністратор).

Мінімальні сутності:


| Сутність     | Опис                                            |
| ------------ | ----------------------------------------------- |
| **User**     | Користувач системи (клієнт або адміністратор)   |
| **Resource** | Ресурс для бронювання (кабінет / кімната / зал) |
| **Booking**  | Бронювання з часовим слотом і статусом          |


## Загальна термінологія

Терміни, які вводяться та використовуються протягом курсу:


| Термін                                   | Опис                                                                              |
| ---------------------------------------- | --------------------------------------------------------------------------------- |
| **Шари (Layers)**                        | Presentation, Application, Domain, Infrastructure                                 |
| **Доменна модель (Domain Model)**        | Anemic vs Rich підходи до моделювання бізнес-логіки                               |
| **Domain Factory**                       | Фабрика для створення доменних об'єктів із перевіркою інваріантів                 |
| **DIP (Dependency Inversion Principle)** | Принцип інверсії залежностей — залежності направлені до домену                    |
| **CQS (Command-Query Separation)**       | Розділення операцій на команди (зміна стану) та запити (читання)                  |
| **Синхронна / асинхронна комунікація**   | Способи взаємодії між компонентами системи                                        |
| **Bounded Context**                      | Обмежений контекст — межі відповідальності модуля                                 |
| **ACL (Anti-Corruption Layer)**          | Антикорупційний шар — захист моделі від зовнішніх змін                            |
| **Domain Events**                        | Доменні події для комунікації між контекстами                                     |
| **Eventual Consistency**                 | Узгодженість у кінцевому рахунку — модель консистентності для розподілених систем |


## Еволюція архітектури


| Лаба | Архітектура                         | Ключовий інсайт                                                                             |
| ---- | ----------------------------------- | ------------------------------------------------------------------------------------------- |
| 1    | CRUD «як вийде»                     | Працює, але бізнес-логіка розмазана, тести залежать від інфраструктури                      |
| 2    | Шарова архітектура + Domain Model   | Бізнес-логіка ізольована, можна змінювати БД/фреймворк, тести домену швидкі                 |
| 3    | CQS                                 | Читання і запис — різні задачі з різними вимогами, простіше розширювати й тестувати         |
| 4    | Синхронна та асинхронна комунікація | Компоненти спілкуються через чітко визначені контракти, система стає гнучкішою              |
| 5    | Модульний моноліт                   | Контексти ізольовані, зміни в одному модулі не ламають інші, система готова до декомпозиції |
